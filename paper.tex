\begin{abstract}
Planning records define restrictions over geographic areas, but source documents often provide only indirect spatial evidence rather than ready-to-use geospatial boundaries. We introduce \textbf{Plan2Map}, a public, manually reviewed benchmark for planning-record PDF-to-GeoJSON digitisation based on UK Article~4 Directions. Given only a source planning document, systems must reconstruct a valid geospatial boundary from notice text, schedules, map plates, labels, and boundary annotations. We propose \textbf{GeoPlanAgent}, a document-grounded, geospatial-tool-in-the-loop system that decomposes the task into evidence extraction, localization, map registration, boundary segmentation, projection, and verification. On 208 cases, GeoPlanAgent achieves 0.690 mean IoU and 0.895 median IoU, with 74.5\% of predictions above 0.5 IoU. Diagnostic analysis reveals a strongly bimodal success pattern and identifies localization and boundary extraction as the dominant bottlenecks. Plan2Map provides a concrete testbed for multimodal geospatial reconstruction from public planning records.
\end{abstract}

\section{Introduction}
\label{section:intro}


Planning records such as UK Article~4 Directions \citep{planningdata2026article4direction} define restrictions that apply to buildings or areas. Concrete geospatial boundary data is the operational form of these records: it allows planning systems to check whether a site or application falls within an affected area, compare the restriction with other planning records and maps, and audit or update the record over time. Although public planning-data portals may publish such boundaries as spatial records\footnote{https://www.planning.data.gov.uk/entity/?dataset=article-4-direction-area}, including GeoJSON geometries, this geometry is not available to a system whose input is only the source planning document. The document usually provides only indirect spatial evidence: notice text, schedules, map plates, map labels, and boundary annotations, from which the boundary must be reconstructed.


Recovering these boundaries is not merely a technical exercise. The UK Government's ``Extract initiative'' reflects a wider push for digitalisation of existing planning records \citep{govuk2025extract,colebourn2025extract}. Recent reviews of planning-application delays further underscore the need for efficient planning information flows \citep{weinstein2024UKgov,mhclg2026planningdelays}. Without machine-actionable boundary data, authorities and downstream users must manually interpret documents, maps, and boundary annotations to reconstruct the affected area, a slow and error-prone process that does not scale across local-authority archives.


To study this problem systematically, we introduce \textbf{Plan2Map}, a public, manually reviewed benchmark for planning-record PDF-to-GeoJSON digitisation. Each example provides a source planning document as input and a verified geospatial boundary as the target output, allowing systems to be evaluated on whether they can reconstruct the affected area from the source document. Plan2Map frames this as an indirect geospatial reasoning task: the system must recover a valid GeoJSON boundary from textual, visual, and geographic evidence rather than extract a field already present in the PDF.


As illustrated in Figure~\ref{fig:task-overview}, Plan2Map requires coupled document, visual, and geospatial reasoning. A system must discover the relevant spatial evidence, select the correct map, interpret drawn or hatched boundary symbols, ground the local map against external basemaps, and return a valid geospatial polygon. Planning PDFs make each step difficult: they may contain multiple maps, scanned or rotated pages, low-resolution cartography, ambiguous hatching, disconnected areas, and textual annotations such as exclusions of named addresses. Text extraction, map segmentation, and geocoding each solve only one part of the problem; none alone recovers the final boundary.

This structure motivates \textbf{GeoPlanAgent}, a decomposition-based system for Plan2Map. GeoPlanAgent mirrors the task decomposition: a \emph{Reader} extracts structured spatial evidence and candidate map pages; a \emph{Locate} sub-agent proposes map centres using offline UK geocoders; a \emph{Worker} registers the planning map against a national basemap, segments the affected boundary, and projects it into geospatial coordinates; and an optional \emph{Critic} reviews committed results.

 
Our \textbf{main contributions} are: \textbf{(i)} We introduce \textbf{Plan2Map}, a public, manually reviewed benchmark for planning-record PDF-to-GeoJSON digitisation, built from UK Article~4 source documents and verified boundary geometries; \textbf{(ii)} We define the associated task and evaluation protocol: given only the source planning document, a system must reconstruct a valid geospatial boundary, evaluated by geometric overlap, polygon F1, centroid error, valid-output rate, and cost; \textbf{(iii)} We propose \textbf{GeoPlanAgent}, a decomposition-based system that mirrors the task structure through reading, localization, map registration, boundary segmentation, projection, and verification; and \textbf{(iv)} We provide diagnostic analysis with direct VLM baselines, essential component ablations, and a failure taxonomy, showing that localization and boundary extraction are the dominant bottlenecks.


\section{Related Work}

\paragraph{Tool-augmented agents and UK planning records.}
Tool-augmented methods such as retrieval-augmented generation, ReAct, and Toolformer are relevant as planning-record digitisation often requires OCR, retrieval, geocoding, segmentation, and map alignment rather than single-pass prediction \citep{lewis2020rag,yao2023react,schick2023toolformer}. In the UK context, the government's Extract initiative and related Google descriptions show growing institutional interest in converting planning PDFs and maps into structured data \citep{govuk2025extract,colebourn2025extract,google2025extract}. However, these initiatives are not presented as reusable academic benchmarks with released proposal-document inputs, ground-truth GeoJSON boundaries, output schemas, and scoring functions.

\paragraph{Map digitisation and georeferencing.}
Our task is also related to map digitisation and georeferencing. MapReader provides tools for analysing large map collections through patch classification and text spotting, while recent historical-map vectorisation benchmarks show that converting scanned maps into usable vector geometries remains difficult \citep{wood2024mapreader,chen2024historicalmapvectorization}. This work motivates the boundary-extraction component of our benchmark, but planning review additionally requires checking whether the recovered geometry is consistent with the legal description and policy record.

\paragraph{Multimodal document understanding.}
GeoPlanAgent draws on work in visual document understanding, where models must read document images, exploit layout, and answer grounded questions. DocVQA established document-image question answering, while LayoutLMv3 and Donut represent layout-aware and OCR-free approaches to document understanding \citep{mathew2021docvqa,huang2022layoutlmv3,kim2022donut}. These works are relevant because Article~4 Direction records contain text, layout, signatures, and embedded maps. However, they target generic document tasks rather than linking statutory language to an externally verifiable geographic boundary.

\paragraph{Geospatial vision-language reasoning.}
Geospatial VQA and remote-sensing VLMs address spatial reasoning more directly. RSVQA introduced VQA over remote-sensing images using OpenStreetMap-derived questions, and later work such as EarthVQA and GeoChat extends this line to relational reasoning and grounded geospatial dialogue \citep{lobry2020rsvqa,wang2024earthvqa,kuckreja2024geochat}. EarthWhere further evaluates open-world geolocation with multimodal models \citep{qian2025earthwhere}. These benchmarks test important spatial capabilities, but mainly focus on aerial, satellite, street-level, or natural imagery. Planning records instead require joint reasoning over PDFs, cartographic symbols, authority metadata, legal text, and GeoJSON boundaries.



\paragraph{Planning, legal, and regulatory AI.}
Planning and legal AI benchmarks evaluate regulatory knowledge and legal reasoning. UrbanPlanBench tests LLMs on urban-planning concepts and regulations, while LexGLUE and LegalBench provide broader legal-language benchmarks \citep{zheng2025urbanplanbench,chalkidis2022lexglue,guha2023legalbench}. These benchmarks are useful comparators for regulatory validation, but they do not require models to connect a planning document's legal language to a map boundary, location, and external geospatial record.

In summary, prior work covers important parts of the problem, but no existing benchmark jointly evaluates planning-record digitisation and regulatory validation over multimodal planning documents, external geospatial tools, and ground-truth boundary geometries.


Existing benchmarks cover pieces of this transformation but not the full task. Document-understanding benchmarks test whether models can extract or answer questions over visually rich documents, but typically expect textual outputs. Map-digitisation and georeferencing work studies scanned maps, vectorisation, or control-point alignment, but usually assumes that the relevant map is already isolated and does not require reasoning over the surrounding planning record. Geospatial vision-language benchmarks test localization or question answering over maps, satellite imagery, or street-level images, but do not require systems to convert a regulatory PDF into a verified WGS84 boundary. Plan2Map fills this gap by evaluating the complete PDF-to-GeoJSON transformation.



\section{Dataset}
We release a benchmark of 208 UK Article~4 Direction records. Article~4 Directions are legal instruments through which UK local planning authorities withdraw nationally-granted permitted development rights for specified areas, typically to protect conservation areas, restrict change-of-use, or preserve landscape character \citep{mhclg2024planningpracticeguidance}. Each released case bundles the original regulatory PDF, a machine-readable ground-truth boundary in GeoJSON, and a rendered location-map visualisation of the boundary on an OpenStreetMap. To our knowledge, this is the first publicly released corpus that links UK planning-control documents to verified, machine-readable boundary geometries, making it suitable for evaluating systems that must read regulatory text, interpret an associated map, and recover a GIS-ready representation of the planning boundary. The records span 1958--2025 and cover twenty-nine distinct local planning authorities across England. The corpus is drawn exclusively from UK planning material and depends on UK gazetteers and base mapping, hence it should be treated as a UK-specific benchmark.

\subsection{Sources and Construction}
 Raw source data was obtained from open UK planning datasets published under the Open Government Licence. Each raw entry consists of an Article~4 Direction PDF paired with a corresponding GeoJSON polygon retrieved from the same authority record. We then carried out a six-stage curation workflow with quality-control checkpoints at each transition: (i)~\emph{document acquisition} from authorised sources; (ii)~\emph{quality assessment} covering visual inspection, completeness, and authenticity; (iii)~\emph{boundary visualisation}, in which the supplied GeoJSON polygon is rendered on an OpenStreetMap to produce the per-case location-map PNG and to enable side-by-side comparison against the boundary drawn in the PDF; (iv)~\emph{metadata annotation} of thirteen structured fields; (v)~\emph{validation} by visual shape matching and cross-reference against the source record; and (vi)~\emph{final quality control} through peer review and discrepancy documentation. The pipeline reduced 270 raw entries to 208 release-ready cases. Specifically, we removed 40 records whose PDF boundary and supplied GeoJSON disagreed (shape mismatches, split maps, or polygons covering only part of a multi-map PDF), 9 records whose PDFs contained no boundary map, and 7 duplicates; we further merged 5 multi-geojson groups (consolidating 11 source rows into 5 cases) where the same PDF was distributed across several authority entries.

\subsection{Dataset Composition}
 Each released data item bundles three artefacts. The \textbf{PDF} contains the regulatory text, schedule of withdrawn permitted development rights, boundary map, and authority signatures. The \textbf{GeoJSON} is a single \texttt{Feature} carrying a \texttt{Polygon} or \texttt{MultiPolygon} geometry in WGS84, representing the application boundary. The \textbf{location-map PNG} overlays the GeoJSON on OpenStreetMap with WGS84 axes for human-readable inspection.
 
The accompanying metadata table records, for each case, an administrative area (county or local authority), a free-text site description, a categorical boundary shape, a document-quality label, a shape-complexity label, and a validation flag. Boundary-shape categories are \emph{rectangular}, \emph{quadrilateral}, \emph{triangular}, \emph{irregular polygon}, \emph{L-shaped}, \emph{elongated strip}, \emph{multiple separate polygons}, and \emph{multiple complex irregular polygons}. Document quality is annotated as \emph{good} for clear text and legible maps, and \emph{bad} for poor scans, faded text, or unclear boundaries. Shape complexity is annotated as \emph{easy} (fewer than eight vertices), \emph{medium} (8--20), or \emph{hard} (more than twenty, or multiple disjoint components). The \texttt{Shape\_Matches} flag, with values \{\emph{Yes}, \emph{Yes--almost}\}, records whether the supplied GeoJSON visually matches the boundary drawn in the PDF; 96.6\% of released cases are exact matches and 3.4\% near matches, with all non-matching cases removed at curation. Identification fields maintain referential integrity between the PDF, the GeoJSON, and the metadata table (Appendix~\ref{appendix:idfields}).


\subsection{Distribution}
 
The dataset spans 1958--2025 with a median document year of 1997; cases are distributed across the 1970s (20.2\%), 1980s (18.3\%), 1990s (12.5\%), 2000s (17.3\%), 2010s (25.0\%), and 2020s (4.8\%), with the remaining 1.9\% pre-1970. Geographically, the corpus covers eighteen counties or county-level administrative areas, concentrated in Kent (16.8\%), Norfolk (15.4\%), Hertfordshire (14.9\%), and London (13.9\%), with the remaining 38.9\% distributed across fourteen further areas, including Cambridgeshire, Staffordshire, Bristol, Leicestershire, Surrey, and Lancashire. At the local-authority level, the five largest groupings are St~Albans (28), Dover District (25), South Norfolk (23), South Staffordshire (13), and Bristol (12). Authority frequency reflects publication practices in the underlying datasets rather than a systematic geographic stratification.

\section{GeoPlan Agent}
\label{section:geoplanagent}

GeoPlanAgent decomposes the boundary-extraction task into a pipeline of three LLM agents and an optional fourth (\Cref{fig:pipeline-overview}). The \emph{Reader} is a single-shot multimodal call that turns a planning PDF into a structured \texttt{PDFInfo} record. The \emph{Worker} is a tool-calling loop that registers the planning map against Ordnance Survey basemap tiles by orchestrating four tools (\Cref{tab:worker-tools}); within \texttt{match\_at} a LoRA-fine-tuned SAM~3 produces the boundary mask and the recovered affine projects it to GeoJSON, so the agent never leaves the loop without a polygon. One of the worker's tools, \texttt{propose\_centers}, delegates to a \emph{Locate sub-agent} that picks a single map centre by calling six offline UK geocoders. An optional independent \emph{Critic} (\Cref{subsection:critic}) reviews the worker's commit against all stored match attempts and may issue a directive to switch candidate or relocate. All four agents share the same multimodal LLM (Gemini~3~Flash by default~\cite{google2025gemini3}), and all four outputs are enforced by pydantic schemas, so the model physically cannot return free text in any phase.

\paragraph{Problem.}
Given a planning PDF $D$ and ground-truth GeoJSON $G_{\text{true}}$, the system emits a predicted GeoJSON $G_{\text{pred}}$. We score $G_{\text{pred}}$ against $G_{\text{true}}$ with IoU and a precision/recall/F1 triple over their polygon set-operations (definitions and headline metric in \S\ref{subsection:summarize}); we additionally report centroid positioning error in metres.

\subsection{Reader}
\label{subsection:reader}

The reader is a single Gemini~3~Flash call (temperature~0) on the PDF binary together with per-page text from a PyMuPDF\,/\,macOS\,Vision\,/\,PaddleOCR cascade (\Cref{appendix:text-extraction}). The prompt tells the model to trust the OCR text block over re-reading the rasterised image for any exact-string field (postcodes, grid references, addresses, scale) because OCR returns verbatim characters whereas vision-only re-reading often substitutes visually similar digits. The output is a \texttt{PDFInfo} object with fields covering site-specific signals (postcodes, OS grid references, site address, road and place names, house-number/road pairs, parish names, visible map labels, adjacency hints, directional modifiers, most-likely town) and document-level metadata (printed scale, page count, one \texttt{MapPageMeta} per map-like page with \texttt{category}, \texttt{area\_group}, \texttt{boundary\_clarity}, \texttt{detail\_level}, \texttt{area\_signature}, local-authority, \texttt{admin\_region}, and an \texttt{is\_district\_wide} flag). \texttt{map\_pages} is the reader's ranked list of \texttt{category=match} pages; the page-rotation needed to make a sideways-laid map upright is decided downstream by a ResNet50 classifier (\Cref{subsection:rotation-classifier}), not by the LLM.

\subsection{Locate sub-agent}
\label{subsection:locate-sub}

The worker's \texttt{propose\_centers} tool delegates to a separate Gemini~3~Flash agent that sees the rendered planning map and the \texttt{PDFInfo} and returns one \texttt{LocatePick}: a centre coordinate, uncertainty radius $\sigma$, confidence label, and short evidence string. The sub-agent has six offline geocoder tools backed by authoritative UK gazetteers: \texttt{postcode} (OS Code-Point Open~\cite{os_code_point_open}, sub-100\,m); \texttt{grid\_ref} (OS BNG, precision-aware); \texttt{place} (OS Open Names~\cite{os_open_names}); \texttt{road} (OML road centroid, optionally LA-filtered); \texttt{intersect} (geometric junction of 2--3 OML roads); \texttt{la\_check} (point-in-polygon against OS BoundaryLine~\cite{os_boundary_line}). Its protocol scans \texttt{PDFInfo} in decreasing order of specificity (site postcode $>$ grid reference $>$ house-number+road $>$ landmark $>$ road $>$ parish $>$ LA centroid), letterhead-checks non-site postcodes via \texttt{la\_check} (dropping anything more than 5\,km from the named LA as council-letterhead noise), and clusters its pool down to one pick. $\sigma$ is set by the picked source: $\approx 200$\,m for multi-source agreement within 500\,m, 300--500\,m for a clean single signal, 800--1500\,m for an ambiguous single signal, and the LA polygon's own radius when nothing better exists. A geocoder-call budget of eight per case and an emergency LA-centroid fallback guarantee that the worker always receives exactly one candidate. When the worker re-invokes the sub-agent with a \texttt{match\_context} string after a weak match, the sub-agent receives both the feedback and its own prior message history and is prompted to pick from a different signal type.

\label{subsection:worker}

The worker is a pydantic-ai loop whose output is a \texttt{BoundaryOutcome} (\texttt{status}\,$\in$\,\{\texttt{accepted}, \texttt{district\_lookup}\}, reasoning). Its four-tool surface is in \Cref{tab:worker-tools}; SAM~3 segmentation and the WGS84 re-projection are not separately addressable, they run automatically inside \texttt{match\_at} and \texttt{commit\_match} so the worker cannot terminate with a positioned map but no polygon. The canonical flow is $\texttt{propose\_centers}\rightarrow\texttt{match\_at}\rightarrow\texttt{commit\_match}$, with the prompt-encoded decision rules: \emph{strong} matches (total inliers $\geq 50$ \emph{and} scale-consistency $\geq 0.8$) commit immediately; everything below \emph{borderline} mandates at least one more \texttt{propose\_centers} call before commit; scale-consistency below 0.5 is overruled when inliers are strong ($\geq 80$); after two attempts the worker picks the highest-total-inlier candidate. An output validator re-prompts the model if \texttt{status="accepted"} arrives without a committed polygon, or if \texttt{status="district\_lookup"} arrives without a successful \texttt{lookup\_district}. The worker is forbidden from inventing coordinates not produced by \texttt{propose\_centers}, capped at five \texttt{match\_at} calls per case, and given a transcript-window of four recent images to keep token cost flat (older images are stripped from the history processor).

\paragraph{Multi-area documents.}
A planning PDF can cover multiple disjoint geographic areas (\emph{Schedule-2} class restrictions, multi-site Article~4 directions). The reader assigns an \texttt{area\_group} index to each match page, and \texttt{match\_at} internally runs MINIMA at the same locate centre on the primary page of \emph{every} \texttt{area\_group}, then unions the resulting polygons into one \texttt{MultiPolygon}. The worker passes a \texttt{page} argument identifying \emph{which} area\_group it wants the named centre evaluated against; alternates are picked automatically.

\paragraph{Matching (\texttt{match\_at}).}
For each area\_group the tool fetches an offline rendering of OS Open Zoomstack~\cite{os_open_zoomstack} tiles centred on the candidate (\Cref{subsection:tile-rendering}), resizes the planning-map crop so one map pixel matches one tile pixel at the chosen Web-Mercator zoom (with $\pm15\%$ scale perturbations to absorb DPI errors), and slides the crop across the tile canvas. At each window MINIMA~\cite{ren2025minima}\textemdash a transformer-based dense-feature matcher built on the LoFTR~\cite{sun2021loftr} architecture and trained for cross-modal map-to-tile matching\textemdash returns sets of corresponding keypoints whose 2$\times$3 affine is then estimated via RANSAC~\cite{fischler1981ransac} with a 10-pixel reprojection threshold. We default to a 4-DOF similarity transform and accept a 6-DOF full affine fallback only when it yields at least 30\% more inliers \emph{and} passes a geometric-sanity check (positive determinant, $|b + c| < 0.15$ on the off-diagonals, aspect ratio above 0.85, average scale within $[0.3, 3.0]$). A Delaunay-consistency filter~\cite{pierdicca2025delaunay}\textemdash building a Delaunay triangulation on the inlier keypoints and discarding triangles whose source/target area ratio falls outside $[0.5, 2.0]$\textemdash is then re-applied, kept only when at least one third of the original inliers survive. Each window receives a vanilla metric $V$ proportional to its (rotation-penalised) inlier count and scale agreement; we retain the best window per zoom and re-rank survivors by the composite score in \Cref{eq:composite-window}. The agent receives total and per-area\_group inlier counts, the road-name-agreement axis (fuzzy-matched against OS Zoomstack road labels within 1.5\,km), and the scale-consistency axis.

\paragraph{Boundary segmentation.}
SAM~3~\cite{meta2025sam3} segmentation runs lazily inside \texttt{match\_at} on first need per page (text prompt fixed to the LoRA-trained anchor ``planning boundary'') and is cached per page across attempts. The cached mask is post-processed through a three-step cleanup chain (\Cref{subsection:mask-postproc}) and projected through the committed affine into tile-canvas pixels and onward to WGS84 via the inverse Web-Mercator transform. For \texttt{is\_district\_wide=True} documents the worker bypasses positioning entirely and calls \texttt{lookup\_district}, returning the offline-stored polygon from OS BoundaryLine.

\paragraph{Composite window score.}
Vanilla MINIMA inlier counts are degenerate against two frequent failure modes: keypoints clustering in one map quadrant and matches landing kilometres from the geocoded anchor on similar urban texture. We address both by composing $V$ with a quadrant-coverage factor $Q\in\{0,\dots,4\}$ (number of map quadrants containing at least one inlier) and a softmax-like distance penalty:
\begin{equation}
  \text{score} = V \cdot \frac{Q}{4} \cdot \frac{1}{1 + d_{\text{km}}}
  \label{eq:composite-window}
\end{equation}
where $d_{\text{km}}$ is the great-circle distance from the geocoded anchor to the matched centre.

\paragraph{Commit gate.}
\texttt{commit\_match} enforces two gates. The \emph{strict} gate refuses any candidate that failed to produce a usable affine on at least one area\_group, forcing a retry via \texttt{propose\_centers}. The \emph{smart} gate re-ranks all stored attempts by the inlier count weighted by whether the predicted centre falls inside the reader-extracted local-authority polygon:
\begin{equation}
  s_{\text{commit}}(c) = n_{\text{inliers}}(c) \cdot \begin{cases} 1.0 & c \in \text{LA} \\ 0.3 & c \notin \text{LA} \end{cases}
  \label{eq:commit-score}
\end{equation}
If a different stored attempt outscores the worker's pick, the gate redirects the commit. The $0.3\times$ outside-LA penalty is small enough that a much-higher-inlier outside-LA candidate can still win (BoundaryLine is itself imperfect) but large enough to dominate near-ties, which is where wrong-town homonyms typically beat the correct candidate.

\subsection{Critic (optional)}
\label{subsection:critic}

After the worker submits, an optional independent LLM critic (same model, same model-family) reviews up to the top three stored match candidates by inlier count (always including the worker's commit) plus a per-candidate metrics block. For each shown candidate it sees a side-by-side panel of the planning map with the SAM~3 mask overlaid and the OS-tile crop with the projected polygon outlined in red. The critic emits a structured \texttt{CriticDirective}\,$\in$\,\{\texttt{approve}, \texttt{switch}, \texttt{retry\_locate}\} with a reasoning string that must cite a concrete visual feature. A \texttt{switch} re-invokes the worker with a fixed-template instruction to commit a different stored \texttt{candidate\_id}; a \texttt{retry\_locate} re-invokes the worker with an instruction to call \texttt{propose\_centers} with new context and try again. Up to two critic iterations are allowed per case; both the worker's first-commit polygon and the post-critic polygon are returned, so a single benchmark run produces both the no-critic and with-critic IoUs (used in the ablation of \Cref{subsection:abl-critic}).

\subsection{Core Experiments}
\paragraph{Headline.}
\Cref{tab:main-result} reports the production numbers. The worker alone reaches mean IoU 0.680 and median 0.893; with the critic enabled the mean rises to 0.690 (\Cref{subsection:abl-critic}). The distribution is sharply bimodal\textemdash 47.5\% of cases at IoU $\geq 0.9$ and 18.3\% below 0.05 with only $\approx 8\%$ in the $[0.3, 0.7]$ middle band\textemdash so the task in this dataset is mostly ``solve cleanly or fail entirely''. End-to-end wall-clock is $\approx 170$\,s per case in serial, dominated by tile rendering on cold cache and MINIMA inference on warm cache; a full 208-case run costs $\approx \$2.75$ in Gemini Flash API spend.

\paragraph{Predictive value of \texttt{n\_inliers}.}
The strongest deployment-relevant signal is the worker's committed inlier count. Among cases that committed with $n_{\text{inliers}} \geq 50$, 83\% land at IoU $\geq 0.7$; below 25 inliers only 7\% do. This gap is the basis of the worker's \emph{strong-match} commit rule (Section~\ref{subsection:worker}) and of the soft quality gate that flags low-confidence commits without nulling the polygon.

\paragraph{Failure modes.}
Of the 49 cases scoring below IoU 0.3, the dominant failure modes are: (i) \emph{wrong-area positioning} (the locate sub-agent picked a coordinate hundreds of metres or kilometres from the true site, typically a council-letterhead postcode that survived the LA filter, or a homonymous road in the wrong town); (ii) \emph{district-lookup mismatch} (15 cases entered the \texttt{lookup\_district} fallback after positioning failed; 13 of them produced a polygon for the wrong administrative area — this is a worse failure pattern than producing no polygon at all because it fakes success); (iii) \emph{wrong-page selection by reader} (the reader's primary map page was a context overview rather than the parcel-scale site map, so MINIMA matched at the wrong scale); (iv) \emph{correct area, bad mask} (positioning correct but the cached SAM~3 mask covered a title block or scale bar rather than the drawn polygon). No-polygon outcomes account for only 5 of 208 cases\textemdash the validator's ``always emit'' policy has all but eliminated that mode.

\subsubsection{Critic Contribution}
\label{subsection:abl-critic}

The optional independent critic (\Cref{subsection:critic}) is the only architectural variant we can ablate cheaply from a single run, because the critic-enabled benchmark snapshots both the worker's first-commit polygon and the post-critic polygon. The paired numbers in \Cref{tab:main-result} give a $+0.010$ mean IoU lift (worker alone $0.680 \rightarrow$ with critic $0.690$); the larger effect is at the tails\textemdash the critic switches the worker's commit on cases where a higher-inlier alternative was waiting in the candidate store, and triggers a \texttt{retry\_locate} on a small number of cases where every stored candidate visually lands in the wrong region. The critic's failure mode is over-approval: deterministic per-axis metrics correlate only weakly with IoU on borderline cases ($\approx 0.13$ Pearson at $n_{\text{inliers}} \in [25, 50]$), so the critic and worker independently mis-trust the same metrics on the same hard cases. Cost: $\approx \$0.20$ extra per case (two extra critic LLM calls, no extra MINIMA / SAM~3 work).

\paragraph{VLM-Direct Protocol}
\label{subsection:abl-vlm-direct-protocol}

VLM-direct is a single-shot Gemini~3~Flash inference per case: no SAM~3, no MINIMA, no agent loop. The model receives the rendered map image together with a fixed prompt that describes the task and enumerates the boundary styles to recognise (solid lines, dashed, dotted, hatched, semi-transparent fill, opaque fill). The schema enforces output as 0+ polygons of $(y, x)$ integer vertices in $[0, 1000]$\textemdash Gemini's native segmentation convention~\cite{google2025gemini3}. We rasterise the polygons at the source image's native resolution (anisotropic scaling) and compute pixel IoU against the same hand-traced masks as the SAM~3 LoRA pool. Decoding is at temperature 0, 8192-token cap, with the schema enforced via \texttt{NativeOutput} mode to prevent truncation of multi-vertex polygons. Six cases failed on first pass (provider 413 / malformed JSON on large pages); we re-ran exactly those six at JPEG-95, 2048-px longest-side input cap (the polygon coordinate system is normalised, so this does not change the IoU axis), giving a merged 0/211 failure rate. For comparison, Gemini~3~Pro on the same prompt with \texttt{output\_retries=0} loses 30 of 211 cases to schema validation, reaching mean IoU 0.555 on its 181 parseable cases\textemdash but the 14\% selection-bias lift on Pro makes it not comparable to the headline Flash row, which is why Flash is what we report.

\section{Conclusion}

We introduced \textbf{Plan2Map}, a public benchmark of 208 manually reviewed UK Article~4 Direction records that pair a regulatory PDF with a verified GeoJSON boundary and a rendered location-map visualisation, and we framed planning-record digitisation as an indirect geospatial reconstruction task in which the target boundary must be recovered from textual, visual, and geographic evidence rather than read off a field in the PDF. To address the benchmark, we proposed \textbf{GeoPlanAgent}, a decomposition-based system that mirrors the task structure through a Reader, a Locate sub-agent backed by offline UK geocoders, a Worker that registers the planning map and segments the boundary, a projection step, and an optional Critic. On the 208-case benchmark the system reaches mean IoU 0.690 with the critic enabled, a median IoU of 0.895, and 74.5\% of cases at IoU $\geq 0.5$. Component ablations confirm that the LoRA-fine-tuned segmenter is essential\textemdash a direct-VLM baseline scores 0.479 mean pixel IoU against the segmenter's 0.892\textemdash and that the dominant residual failures are concentrated in localisation and reader page-selection rather than in segmentation. Plan2Map and GeoPlanAgent together provide a starting point for systematic study of PDF-to-GeoJSON digitisation, and we expect future work to focus on more robust localisation, better discriminators in the borderline regime, and extension of the benchmark to additional jurisdictions.

\section*{Limitations}

The dataset is UK-specific and over-represents London, Kent, and Norfolk\textemdash 67\% of cases come from just three of the 15+ local authorities sampled. Pipeline components that depend on the UK geocoder stack (Code-Point Open, OS Open Names, OS BoundaryLine, OS OpenMap Local, OS Open Zoomstack) do not transfer to other jurisdictions without re-pointing each tool at the local equivalent; the agent's framing should transfer, but we cannot empirically verify this without an analogous dataset for at least one other country.

The boundary-segmentation step assumes that the application boundary is visually drawn on the planning map. Documents whose application area is described purely by text (e.g. ``land north of 14 Manor Road'') with no overlaid boundary line fall outside the pipeline's competence\textemdash the Locate sub-agent will still produce a centre, but SAM~3 cannot segment a polygon that does not exist on the page.

The LA polygon used in the smart-commit gate (\Cref{eq:commit-score}) comes from OS BoundaryLine, which is updated semi-annually. A small number of cases\textemdash typically late-1970s and early-1980s Article~4 directions that pre-date several boundary revisions\textemdash sit just outside the current BoundaryLine polygon for their stated LA, triggering the outside-LA penalty incorrectly. We retain the case in the benchmark but the metric over-penalises the pipeline on those documents.

SAM~3 fine-tuning was performed on CUDA with bf16 mixed precision; validation (the per-fold numbers in \Cref{tab:sam3-cv}) is scored on Apple Silicon with fp16 autocast. A small amount of run-to-run nondeterminism in the MPS evaluation kernels means the cross-validation numbers are reproducible to $\pm 0.003$ rather than bit-exact. The dominant published-source bottleneck is therefore not training stochasticity but the size of the annotated training pool (211 cases), which we expect any future extension of this dataset to relax.

\section{Implementation Details}
\label{sec:impl-appendix}

This appendix expands the per-component summary in the main body. Subsections follow data-flow order.

\subsection{Text Extraction}
\label{appendix:text-extraction}

Per-page text is produced by a three-step cascade. PyMuPDF reads the embedded text layer for born-digital pages; if its output for a given page is shorter than 50 characters we fall back to OCR. The primary OCR backend is the macOS Vision framework (Apple Neural Engine accelerated, $\approx 0.5$--2\,s per page on Apple Silicon, accurate-level recognition with language correction over \texttt{en-GB}/\texttt{en-US}); PaddleOCR (3.x API, confidence floor 0.5) serves as a cross-platform fallback for environments without Vision or when Vision returns empty on a particular page. OCR rasterisation is at 250\,DPI. Per-page text is capped at 8\,000 characters and the entire cascade output is cached on disk keyed by PDF content hash, so reruns are instant.

\subsection{Auto-Rotation Classifier}
\label{subsection:rotation-classifier}

Many UK planning maps are laid out sideways on the page. We replace the agent's earlier visual rotation decision (which destroyed several cases by rotating documents that were already upright) with a learned 4-class ResNet50~\cite{he2016resnet} classifier over $\{0°, 90°, 180°, 270°\}$. Training data is generated by applying each of the four rotations to 211 maps confirmed upright by hand, yielding 844 samples; the split is by case using the same deterministic 5-fold assignment as the SAM~3 fine-tune so all four rotations of a given case fall in the same partition. The backbone is ImageNet-pretrained ResNet50 (V2 weights), full fine-tune at $768\times768$ input, ColorJitter + RandomErasing augmentation, AdamW (lr $10^{-4}$, weight-decay $10^{-4}$), cosine schedule, gradient clipped at norm 1.0, early stop patience 4. At inference we apply 4-rotation TTA\textemdash predicting on the input plus three CW rotations and cyclically shifting each prediction back to the original frame before averaging the softmax\textemdash and abstain (return zero rotation) whenever the top-class probability falls below 0.50.

\subsection{Matching (\texttt{match\_at}) Details}
\label{subsection:matching}

For each candidate centre the planning map is resized so that one map pixel corresponds to one tile pixel at the chosen Web-Mercator zoom; the best zoom is computed from the printed scale and the rendering DPI as $z = \log_2(C \cos\phi / m_{\text{px}})$, where $C \approx 156\,543$\,m is the equator pixel scale at zoom 0 and $\phi$ is the candidate latitude (zoom range 15--19 in practice). MINIMA~\cite{ren2025minima} (LoFTR~\cite{sun2021loftr} backbone, confidence threshold 0.2) is invoked at the chosen zoom plus its two neighbours and at $\pm15\%$ scale perturbations of the chosen mpp. The window stride targets $\approx 100$ pixels per step with a 32-pixel floor. RANSAC affine recovery uses 4-DOF (\texttt{cv2.estimateAffinePartial2D}) at a 10-pixel reprojection threshold; the 6-DOF (\texttt{cv2.estimateAffine2D}) fallback is committed only when its inlier count is at least $1.3\times$ the 4-DOF count and the recovered affine has positive determinant, $|b + c| < 0.15$ on the off-diagonals, aspect ratio above 0.85, and average scale within $[0.3, 3.0]$. The Delaunay-consistency filter~\cite{pierdicca2025delaunay} is then applied: for each Delaunay triangle on the inlier points we compute the source/target area ratio and discard triangles outside $[0.5, 2.0]$; the affine is re-fitted on the surviving inliers, reverting to the original RANSAC fit when the filter would leave fewer than $\max(8, n_{\text{inliers}}/3)$ points.

\subsection{Tile Rendering}
\label{subsection:tile-rendering}

Tiles are rendered offline from the OS Open Zoomstack~\cite{os_open_zoomstack} GeoPackage ($\approx 750$\,MB, OGL v3) by a styler that approximates UK planning-map basemaps: salmon-pink (\#FFB3B3) buildings with grey outlines, white road fills with grey casings, light green woodland, light blue surface water, dark grey railways. Road widths scale with zoom (a Local Street is 1.5\,px at $z=17$, proportionally wider at higher zoom); roads draw casings first then fills so minor roads are correctly subordinated to major. Geometry is projected from BNG (EPSG:27700) through Web-Mercator (EPSG:3857) to per-tile pixels with a 5\% bbox inflation. Rendered tiles are cached to disk keyed by zoom and BNG tile index; the first call for a case takes $\approx 5$--20\,ms per tile, subsequent calls are sub-millisecond reads.

\subsection{SAM~3 Fine-Tuning}
\label{subsection:sam3-finetune}

SAM~3~\cite{meta2025sam3} is fine-tuned with rank-16 LoRA~\cite{hu2022lora} adapters on every $\{q, k, v, o\}$ attention projection and the two MLP projections ($\text{fc}_1$, $\text{fc}_2$) of every transformer subsystem (vision encoder, text encoder, geometry encoder, DETR encoder/decoder, mask decoder)\textemdash deliberately broad, since restricting LoRA to the mask decoder alone produced markedly worse instance candidates in earlier ablations. The mask embedder, presence head, and semantic projection layer are fully trainable via PEFT's \texttt{modules\_to\_save}. LoRA $\alpha = 32$, dropout 0.05, bias \texttt{none}. The query phrase is fixed to ``planning boundary'' (the LoRA-trained anchor). Training jointly optimises the semantic and instance heads:
\begin{align}
  \mathcal{L}_{\text{sem}} &= 5\,\mathcal{L}_{\text{focal}}^{\alpha=0.6,\gamma=1.6} + 5\,\mathcal{L}_{\text{dice}} + r(e) \cdot 0.5\,\mathcal{L}_{\text{surf}} \nonumber \\
  \mathcal{L}_{\text{inst}} &= 5\,\mathcal{L}_{\text{focal}}^{\alpha=0.25,\gamma=2} + 5\,\mathcal{L}_{\text{dice}} + 2\,\mathcal{L}_{\text{cls}} + \mathcal{L}_{\text{pres}} \nonumber \\
  \mathcal{L} &= \mathcal{L}_{\text{sem}} + \mathcal{L}_{\text{inst}}
  \label{eq:sam3-loss}
\end{align}
where $r(e) = \min(1, e/15)$ ramps the surface loss linearly over the first 15 epochs (the surface term penalises sigmoid prediction mass at signed-distance pixels far from the boundary). The instance focal+dice losses are computed on the slot selected by a Hungarian-style match with cost $-\text{IoU} - 0.05\,\sigma(\text{cls}_i)$. The classification loss is per-slot focal with a soft positive target $\sigma(\text{cls}_{\text{best}})^{0.25} \cdot \text{IoU}_{\text{best}}^{0.75}$ on the matched slot and zero elsewhere\textemdash this prevents the cls head from saturating on partially-correct masks while still anchoring slot identity across epochs. The presence BCE has target $= 1$ everywhere since every training image contains a planning boundary. Optimiser is AdamW (lr $2 \times 10^{-4}$, cosine decay to 5\% of base, weight-decay 0.01), batch size 1 with gradient accumulation 4, gradient clipping 0.1, bf16 mixed precision on CUDA, max 20 epochs, early-stop patience 6 epochs on semantic-head validation IoU.

\paragraph{$k$-fold routing.}
5-fold CV uses a deterministic longest-processing-time-first bin-packing assignment that preserves ``stay-together'' case groups (multi-page renders from a single source and twin sets of cases that share the same planning site are forced into the same fold). At inference the model loader keeps each fold's adapter on disk and \texttt{set\_fold\_for\_case} activates the LoRA from the fold that excluded the current case (training cases) or an MD5-hash assignment (deployment cases not in the training pool). This eliminates GT leakage when benchmarking on cases that were part of the SAM~3 training pool and lets us report results on the entire labelled set.

\paragraph{Style-transfer augmentation.}
To increase effective training diversity from a small annotated pool, every training sample is re-rendered (50\% probability) in one of eight randomly-chosen boundary styles: solid outline (3--10\,px), thick outline (10--20\,px), thin outline (1--3\,px), dashed, dotted, semi-transparent fill ($\alpha \in [0.2, 0.5]$), opaque fill, or diagonal hatching. Eleven colours are sampled uniformly. The existing boundary is faded (desaturate 70--100\%, blend toward local background mean) before the new style is drawn, and the drawn boundary is roughened with Gaussian blur, additive noise, and morphological roughening to mimic scan artefacts.

\subsection{Mask Post-Processing}
\label{subsection:mask-postproc}

The raw SAM~3 mask is run through a three-step cleanup chain before contour extraction. \texttt{keep\_dominant\_components} drops connected components below 5\% of the largest component's area (100-pixel absolute floor). \texttt{expand\_thin\_mask} detects masks whose foreground covers less than 10\% of their own bounding box and dilates them with an elliptical kernel sized to $\approx 1.5\%$ of the bounding box's smaller dimension, accepting the result only when fill stays under 50\% of the image. \texttt{fill\_mask\_holes} morphologically closes interior gaps with an ellipse kernel sized to 1\% of the smaller image dimension and fills remaining holes by drawing external contours filled. The cleaned mask is then converted to GeoJSON: external contours larger than 100 pixels are simplified by Douglas--Peucker ($\varepsilon = 3$\,px), projected through the committed affine into tile-canvas pixels, and converted to WGS84 via the inverse Web-Mercator. Polygons are explicitly closed and emitted as \texttt{Polygon} (one contour) or \texttt{MultiPolygon} (several).

\subsection{Independent Critic}
\label{subsection:critic-appendix}

When enabled, the critic runs after the worker submits. It is a pydantic-ai \texttt{Agent} (same model family as the worker) whose output is a \texttt{CriticDirective}. Inputs are the top-3 stored candidates by total inliers (always including the worker's commit even when its inlier count puts it outside the top-3) sent as separate full-resolution images plus a text block listing per-(candidate, area\_group) metrics ($n_{\text{inliers}}$, road-name agreement, scale consistency). Each candidate image is LEFT|RIGHT: planning map with SAM~3 mask overlaid in translucent green on the left, OS-tile crop with the projected polygon outlined in red on the right. The critic must cite a concrete visual feature in its reasoning (named road, building block, junction) and outputs one of three actions: \texttt{approve}, \texttt{switch} (specifying \texttt{chosen\_candidate\_id}), or \texttt{retry\_locate}. A \texttt{switch} re-invokes the worker with the instruction to call \texttt{commit\_match} on the named candidate; a \texttt{retry\_locate} re-invokes the worker with the instruction to call \texttt{propose\_centers} with a \texttt{match\_context} string. Up to two iterations are allowed; if the geojson does not change between an action and the next state snapshot, the loop exits. The worker is opaque to the critic\textemdash its system prompt is never modified, and the re-invocation messages use neutral framing rather than naming a ``reviewer''.

\subsection{Training Dataset Construction}
\label{appendix:dataset-prep}

The SAM~3 fine-tuning dataset is constructed from a three-stage hand-annotation pipeline. First, each evaluation PDF is rendered at DPI 200 without auto-rotation (so the annotator works in the raw render frame) and seeded with an initial polygon obtained by projecting the ground-truth GeoJSON through a cached affine from an earlier benchmark run. Second, a single annotator reviews each pre-rendered case in a browser-based Flask UI and either accepts the seeded polygon, edits its vertices to match the boundary actually drawn on the page, or rejects the case (typically when the PDF contains no usable map). Third, curated annotations are copied into the training dataset with deterministic 5-fold assignment by the bin-packing scheme above. The final training pool contains 211 annotated maps; the per-fold split is approximately 42--43 validation cases per fold.

\subsection{System Prompts}
\label{appendix:prompts}

The four LLM agents each have a fixed system prompt enforced together with a pydantic schema for the output. Verbatim prompts are listed below (whitespace condensed for typesetting); the schemas and field-level descriptions are in \texttt{tools/agent/schemas.py}.

\paragraph{Reader prompt (\texttt{READER\_SYSTEM\_PROMPT}).}
\begin{quote}\small\sloppy
``You are a UK planning document reader. Read every page of the PDF carefully and populate the \texttt{PDFInfo} schema. \emph{[Field-by-field guidance:]}\\
\textbf{map\_page\_details:} one \texttt{MapPageMeta} entry per page that contains any map-like content; mark it \texttt{category=`match'} only if it is a real positionable map with a drawn planning boundary on a cartographic background; otherwise \texttt{`discard'}. Discard aggressively\textemdash false discards are worse than false matches. \texttt{area\_group=-1} for discards; for matches, pages of the same geographic area share the same integer (\texttt{0, 1, 2, \dots}); pages of the same area shown for different Schedule-2 class restrictions share the same area\_group and identical \texttt{area\_signature}. \texttt{boundary\_clarity} is \texttt{`clear'} only when the boundary is unambiguous to trace AND cartographic detail is visible around it; otherwise \texttt{`ambiguous'}. \texttt{detail\_level} is one of \texttt{close} (parcel), \texttt{medium} (neighbourhood), \texttt{wide} (town).\\
\textbf{map\_pages:} ranked list of \texttt{category=`match'} page numbers. Within an area\_group, rank by boundary\_clarity, then wider detail\_level, then more cartographic detail. Do not rank by recency, DPI, or annotation density. Ordering across area\_groups is arbitrary.\\
\textbf{postcodes / grid\_refs:} extract every UK postcode and OS grid reference, including from form fields and map title blocks.\\
\textbf{is\_district\_wide / district\_name:} \texttt{true} when the boundary covers an entire borough/district/ward/parish/conservation area; \texttt{district\_name} is the standard UK admin name with a `UK' suffix, optionally with `\textbar' alternates.\\
\textbf{site\_address:} the SITE address (not council/agent/architect office addresses).\\
\textbf{directional\_modifier:} compact directional phrase if present (`Field north of 98 Pipers Lane'); null when ambiguous.\\
\textbf{house\_number\_road\_pairs:} `<numbers> <road>'; collapse ranges (`126, 128, 130, 132 and 134 Norwich Road' $\rightarrow$ `126-134 Norwich Road'). Not parcel numbers.\\
\textbf{parish\_names / admin\_region:} bare names.\\
\textbf{likely\_town\_or\_city:} synthesised best guess from all signals; crucial for disambiguating homonymous road names.\\
\textbf{visible\_map\_labels:} labels printed on the map image itself (copy verbatim).\\
\textbf{adjacency\_hints:} named features explicitly adjacent (`adjoining X', `bordered by Y').''
\end{quote}

\paragraph{Worker prompt (\texttt{WORKER\_SYSTEM\_PROMPT}).}
\begin{quote}\small\sloppy
``You are the worker agent in a pipeline that extracts the application-site boundary from UK planning PDFs and projects it to a WGS84 GeoJSON polygon. \emph{[\dots]} Your tool surface is \texttt{propose\_centers} $\rightarrow$ \texttt{match\_at(page=N, \dots)} $\rightarrow$ \texttt{commit\_match} $\rightarrow$ return \texttt{BoundaryOutcome}, plus \texttt{lookup\_district} for documents covering an entire administrative area. SAM3 segmentation and GeoJSON projection are automatic.\\
\textbf{Workflow:}\\
1. \texttt{propose\_centers()}\textemdash one ranked candidate (lat/lon/sigma\_m/source).\\
2. \texttt{match\_at(page=N, name, lat, lon)} on that candidate. Returns \texttt{total\_inliers} plus per\_group breakdown ($n_{\text{inliers}}$, road\_name\_agreement, scale\_consistency).\\
3. Decide:\\
\hspace*{1em} STRONG ($n_{\text{inliers}} \geq 50$ AND scale\_consistency $\geq 0.8$) $\rightarrow$ \texttt{commit\_match}.\\
\hspace*{1em} BORDERLINE $\rightarrow$ MANDATORY: try at least one more \texttt{propose\_centers} candidate before committing.\\
\hspace*{1em} scale\_consistency $<$ 0.5 with strong inliers ($\geq 80$) $\rightarrow$ trust inliers.\\
\hspace*{1em} After 2+ \texttt{match\_at} attempts: pick the highest-total-inlier candidate and \texttt{commit\_match}; the call internally re-ranks with a 70\% penalty applied to candidates outside the reader's admin\_region polygon, so you don't need to verify LA containment yourself.\\
4. Submit \texttt{BoundaryOutcome} with \texttt{status=`accepted'} (or \texttt{`district\_lookup'}).\\
Budget: max 5 \texttt{match\_at} calls. No invented coordinates\textemdash every (lat, lon) must come from \texttt{propose\_centers}; to add a missing place, call \texttt{propose\_centers(extra\_terms=[\dots])}. After a weak match, you can re-call \texttt{propose\_centers(match\_context=`\dots')} describing what went wrong, and the locate sub-agent will pick from a different signal type. The pipeline always produces a polygon\textemdash never refuse.''
\end{quote}

\paragraph{Locate sub-agent prompt (\texttt{LOCATE\_SYSTEM\_PROMPT}).}
\begin{quote}\small\sloppy
``You are the LOCATE STAGE \emph{[\dots]}. Given pdf\_info text fields AND the rendered planning map image, produce ONE center coordinate (lat, lon) + uncertainty radius $\sigma$ + confidence. Six offline tools: \texttt{postcode}, \texttt{grid\_ref}, \texttt{place(q, la?)}, \texttt{road(q, la?)}, \texttt{intersect(road\_a, road\_b, la?, road\_c?)}, \texttt{la\_check(lat, lon, la)}.\\
\textbf{Protocol:} (1) View the map carefully for labels missed by the reader. (2) Scan pdf\_info in decreasing specificity: full postcode IN site\_address (= SITE postcode, trust) $>$ grid\_ref $>$ house\_number + named road $>$ named place / landmark from pdf\_info OR map image $>$ road name (LA-filtered) $>$ parish $>$ LA centroid. (3) Letterhead-check postcodes: for each postcode not in site\_address, \texttt{la\_check}; drop if $>5$\,km from admin\_region. (4) Build a pool of 2--4 candidates from different signal types. (5) Cluster \& pick: 2+ candidates within 500\,m $\rightarrow$ $\sigma=200$, conf=high; clean single signal (SITE postcode, grid\_ref, intersect) $\rightarrow$ $\sigma=300$--500, high; single ambiguous (road, common place) $\rightarrow$ $\sigma=800$--1500, med; LA-only $\rightarrow$ $\sigma$ from tool, low. (6) Validate with \texttt{la\_check}. (7) Emit \texttt{LocatePick} to terminate. Budget: $\leq 8$ geocode calls.''
\end{quote}

\paragraph{Critic prompt (\texttt{CRITIC\_INSTRUCTIONS}).}
\begin{quote}\small\sloppy
``You are an independent reviewer of a UK planning-boundary extraction pipeline \emph{[\dots]}. You see ONE image per candidate (LEFT = planning map with SAM mask in green; RIGHT = OS tile with projected polygon in red) plus a metrics block listing $n_{\text{inliers}}$, road\_name\_agreement, and scale\_consistency per (candidate, area\_group). The committed candidate is tagged in both the panel label and the metrics block. Trace named roads, settlement shapes, building blocks between the planning map and the OS render; the drawn boundary on the planning map should sit where the red outlined polygon sits in the OS render. Decision (pick one): \texttt{approve} (committed candidate is correct), \texttt{switch} (a different stored candidate is visually better; specify it in \texttt{chosen\_candidate\_id}), \texttt{retry\_locate} (none of the candidates show good correspondence\textemdash agent should re-locate from a different geocoding signal). Reasoning MUST cite a concrete observation: a specific feature you saw matched or mismatched, or a specific metric. Never use vague language (`looks fine', `reasonable').''
\end{quote}
